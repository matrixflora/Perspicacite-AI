## Bugs fixed in this audit (committed to `main`)

### 1. Chat router silently routes to DeepSeek regardless of config — FIXED
**File:** `src/perspicacite/web/routers/chat.py` (commit `e3e00d1`)

`_stream_rag_mode` built `RAGRequest` without passing `provider`/`model`,
so every basic/advanced/profound/contradiction chat inherited the
dataclass defaults (`"deepseek"` / `"deepseek-chat"`) and ignored
`llm.default_provider` from `config.yml`. Pure latent bug — manifested
as `AuthenticationError: DeepseekException` on every chat once you
swap the configured provider.

### 2. Embedding provider ignored config when no API fallback wanted — FIXED
**File:** `src/perspicacite/web/state.py` (same commit)

State directly instantiated `LiteLLMEmbeddingProvider` instead of the
`create_embedding_provider()` factory. Two consequences: (a) local
sentence-transformers models (e.g. `all-MiniLM-L6-v2`) couldn't be
used as configured (litellm doesn't route them), (b) API embeddings had
no automatic local fallback even though the code for that existed.

### 3. `non_streaming_pipeline_error` log crashed every non-streaming chat — FIXED
**File:** `src/perspicacite/web/routers/chat.py` (same commit)

`logger.warning("...", error=msg)` against a **stdlib** logger raises
`TypeError` → HTTP 500. (The line was structlog-style.) Replaced with
`%s` formatting.

### 4. MCP `generate_report` had the same DeepSeek-default bug — FIXED
**File:** `src/perspicacite/mcp/server.py` (commit `645a9f4`)

Same bug class as #1 but in the MCP tool. Every MCP-driven generate_report
silently routed to DeepSeek. Caught live by an MCP client connecting to
`/mcp/` and getting `DeepseekException` instead of the configured
Anthropic answer.

### 5. Zotero base_url is hardcoded — local desktop API impossible — FIXED
**File:** `src/perspicacite/integrations/zotero.py` + schema + call sites (commit `6416fa0`)

`ZOTERO_API` was a module-level constant pinned to `https://api.zotero.org`.
Linked-Files / ZotFile / not-synced-to-cloud attachments could never be
ingested — the Web API has no way to serve files that aren't uploaded to
zotero.org. And every build was forced through cloud rate limits even
when the data sits in `~/Zotero/storage` locally.

Now `ZoteroConfig.base_url` is configurable. On loopback the api_key
requirement is also relaxed because Zotero's desktop HTTP server trusts
local connections. User flow:

1. Zotero 7+: Settings → Advanced → "Allow other applications on this
   computer to communicate with Zotero" (Zotero 6: about:config →
   `extensions.zotero.httpServer.enabled = true`).
2. config.yml:
   ```yaml
   zotero:
     enabled: true
     base_url: "http://localhost:23119/api"
     library_id: "<id>"
     library_type: "group"   # or "user"
     api_key: ""             # optional on loopback
   ```

### 25. SI fetch via PMC OA S3 bucket — FIXED
**Files:** `pipeline/download/pmc.py`, `pipeline/download/supplementary.py` (commit `b80edd8`)

Final SI fetch reliability fix. The previous round added a browser
User-Agent + HTML-content-type detection, but pmc.ncbi.nlm.nih.gov
still returns a 1817-byte "verify you are human" HTML interstitial
for many papers regardless. Discovery: PMC's S3 OA bucket
(`pmc-oa-opendata.s3.amazonaws.com/<pmcid>.<version>/`) mirrors the SI
files alongside the JATS XML — same `<href>` values, public read, no
bot-gating, no rate limits. Listed PMC1817752.1/ via S3
ListObjectsV2 and confirmed.

Implementation:
- `_extract_supplementary_from_xml` now composes:
  - `url`: S3 OA bucket (primary)
  - `alt_urls`: [gated pmc.ncbi.nlm.nih.gov URL] as fallback
- `download_supplementary_to_capsule` tries the primary then walks
  `alt_urls`; first non-None response wins.
- Failure summary lists all attempted URLs for debuggability.

Live-verified: 10.1371/journal.pone.0000308 fetched all 4 SI files
(s001.doc, s002.doc, s003.txt, s004.xls — 149 KB total) on first
try via S3. Non-OA papers (e.g. Nature author-manuscripts) still
need the gated URL and may still be bot-blocked — but that's an
external constraint, not silent any more.

### 24. SI fetch: modern PMC URL + bot-gate detection — FIXED
**Files:** `pipeline/download/pmc.py`, `pipeline/download/supplementary.py` (commit `8edfb41`)

Live test of fetch_supplementary against the Nature paper surfaced two
real issues:

a) `_extract_supplementary_from_xml` composed
   `https://www.ncbi.nlm.nih.gov/pmc/articles/<PMCID>/bin/<file>` —
   NCBI migrated to `pmc.ncbi.nlm.nih.gov/articles/instance/<numeric>/bin/<file>`.
   Updated.

b) Even with the right URL, NCBI gates non-browser User-Agents,
   returning either 404 or a 1817-byte HTML "verify you are human"
   interstitial with `content-type: text/html` and HTTP 200.
   `fetch_supplementary_file` now sends a realistic Mozilla User-Agent
   and rejects responses whose content-type starts with `text/html`
   (logging `si_fetch_got_html_likely_bot_gate`) so we don't write
   the HTML page as a fake `.pdf`. Optional
   `expected_mime_prefix` arg for strict callers.

Documented limitation: PMC will still rate-limit / gate from a single
IP within short windows. Future hardening could route through NCBI's
E-utilities API key.

### 20. SI fetcher + Springer/Nature ESM + ACS extractors — NEW FEATURE
**File:** `pipeline/download/supplementary.py` + capsule + MCP (commit `17ee29e`)

Builds on PMC SI manifest (fix #19). Three extractors:
- **PMC** (most reliable; preserves labels + captions)
- **Springer/Nature ESM**: regex `static-content.springer.com/esm/...`
  from the article landing-page HTML
- **ACS supporting info**: scrape `pubs.acs.org/doi/suppl/<doi>` for
  `suppl_file/<filename>` URLs

`discover_supplementary(doi)` is the unified entry point: tries PMC →
Springer → ACS. `fetch_supplementary_file(url, max_bytes=50MB)`
downloads with a hard size cap (SI files can be 100s of MB).
`download_supplementary_to_capsule` enforces per-file + per-record
caps and optionally filters non-text mimes.

New MCP tool: **`fetch_supplementary(kb_name, paper_id)`** — reads
the capsule's `supplementary/index.json`, fetches into
`<capsule>/supplementary/files/`, writes `fetched.json`. 17 MCP tools
total now (was 16). Idempotent (skips if `fetched.json` exists unless
`force=True`).

### 19. SI manifest recording from PMC JATS — NEW FEATURE
**Files:** `pipeline/download/pmc.py`, `pipeline/capsule_builder.py`,
`docs/superpowers/capsule_schema.md` (commits `212b8c8`, `aa441a9`)

Open audit finding N. Capsule build now scans the paper's PMC JATS
XML for `<supplementary-material>` (and `<inline-supplementary-material>`)
entries and writes
`<capsule>/supplementary/index.json` with `{items: [{label, caption,
href, url, mime_type}, …], source: "pmc_jats"}`. URL composition uses
the PMC bin/ mirror so the files are directly fetchable.

Handles xlink:href on the wrapper, plus child `<media>`, `<graphic>`,
`<self-uri>`, `<ext-link>` elements when the wrapper has no href —
necessary for PubMed-reformatted Nature/Cell/etc. papers.

Bytes are not downloaded by default (some SI is large opaque binary);
metadata-only is enough for the capsule reader and external-resource
fetcher to act on. A dedicated `fetch_supplementary` helper is the
obvious next step.

Live-verified:
- Nature thermometry (10.1038/nature12373): 1 PDF
- PLOS One (10.1371/journal.pone.0000308): 4 SI files with proper
  Text S1/S2/S3 labels

Future: Springer/Nature ESM via `static-content.springer.com/esm/...`
URL pattern, ACS supporting-info pages. PMC alone covers the OA tail
that matters most for life-sciences research.

### 9. Zotero 429 silent failure + per-call library_id override — FIXED
**File:** `src/perspicacite/integrations/zotero.py` + tool/route (commit `2495a78`)

### 8. Agentic mode `answer.sources` was empty + paper_id format inconsistency — FIXED
**Files:** `web/routers/chat.py`, `mcp/server.py`, `web/routers/kb.py`,
`cli.py` (commits `26ce023` + `95991fd`)

The orchestrator emits `papers_found` AFTER `answer` (so the browser
can append citations without overwriting innerHTML), but the chat-route
contract puts sources ON the answer. First attempt accumulated papers
BEFORE answer — wrong order, sources stayed empty. Fix: buffer the
answer event when it arrives, flush it once `papers_found` lands (or
at stream-end with empty sources). Verified live: 7 sources in
agentic answer event.

Also: `paper_id` was stored as bare DOI but CLI `--paper` help says
`doi:10.1234/abc` — agents following the CLI convention got "paper not
found" from MCP. Now MCP/REST/CLI all strip an optional `doi:` prefix
before lookup.

Bonus latent bug surfaced and fixed: `MCPState.__init__` was missing
`self.job_registry = None`, so `fetch_paper_resources` over MCP raised
AttributeError as soon as the paper-id normalization let it past the
"not found" gate.

### 7. Profound mode hallucinated meta-essay with 0 sources — FIXED
**File:** `src/perspicacite/rag/modes/profound.py` (commit `012af31`)

The plan-review step (`_review_and_adjust_plan`) runs an LLM evaluator
that sees only the question text + step success flags — never the actual
KB content. For small KBs and broad questions, per-step analyses often
mark early steps as unsuccessful, hitting `max_consecutive_failures`
within the first cycle. The evaluator then reasonably concludes
"unanswerable" because the question lacks context — and the mode
short-circuits to `PROFOUND_UNANSWERABLE_QUESTION_PROMPT_TEMPLATE`,
generating a generic essay about why the question can't be answered.

Reproduced live against `smoke` (1 paper, NV-diamond thermometry):
profound returned a 500-word hallucination about device temperature
coefficients, RTDs, and thermocouples — totally unrelated to the paper.

Fix: when the evaluator recommends `explain_limitations` but ZERO
documents have been retrieved across all completed steps so far, override
and fall through to plan-modification. The evaluator is judging
answerability with no evidence — that's "be cautious about an ambiguous
question" mode, not "give up" mode.

After the fix: same KB + query returns a real grounded answer citing
Kucsko et al. 2013 with the actual "1.8 mK" sensitivity from the paper,
via Early Exit.

### 6. Provenance rows never saved on streaming chats — FIXED
**File:** `src/perspicacite/rag/engine.py` + `web/routers/chat.py` (commit `e80eab9`)

Two compounding bugs broke `GET /api/conversations/{id}/provenance`:

a) `with collecting(collector):` uses `ContextVar.reset(token)` on exit.
   In an async generator consumed by Starlette's StreamingResponse, the
   yield can resume in a different asyncio Context than the one that set
   the token. The reset raises *"Token was created in a different
   Context"* and the `_save_provenance` call after the `with` block is
   silently skipped.

b) Even after fixing (a), the save still only ran on the happy path.
   When the client closes after the final `done` event, the async
   generator gets `aclose()`d — `GeneratorExit` is raised inside the
   `async for`. That's a BaseException, not caught by `except
   Exception`. Result: no save, no error log, silent drop.

Fix: set the ContextVar without resetting (each request runs in its own
task; the var goes out of scope with the task). Move the save into a
`finally:` block with a `saved` flag so it runs once whether the stream
completes, raises, or is closed early.

Verified live: `conversation_id` is now persisted on basic/advanced rows
and `GET /api/conversations/{id}/provenance` returns the full
`retrieval_events` + `llm_calls_index` trace.

### F. Zotero rate-limit (HTTP 429) errors are swallowed silently

When the BioMedOmicsAI build's fan-out of `/items/{key}/children` calls
hit Zotero's per-account rate limit, a subsequent MetaboLinkAI build:
1. Created the empty KB ✓
2. Got 429 on every Zotero API call
3. **SSE stream emitted `{"type":"done","result":{"per_kb":[{"kb_name":"...","added_papers":0,"added_chunks":0,"skipped":0}]}}`** — i.e. reported "success" with zero papers.

No error, no retry, no warning to the user. Should at minimum:
- Detect 429 + 5xx and emit a `{"type":"error","kind":"rate_limit"}` SSE event
- Surface a retry-after backoff (Zotero returns `Retry-After` header)
- Not create the empty KB if no papers were ingested (or roll it back)

### G. `build_kbs_from_zotero` reads single `library_id` from config — no per-call override

The current tool/route reads `zotero.library_id` from `config.yml` at startup.
To cover N group libraries (e.g. BioMedOmicsAI + MetaboLinkAI) requires:

```
edit config.yml → library_id: 5691738 → restart → build
edit config.yml → library_id: 5453037 → restart → build
```

This is friction-rich for the "different KB per Claude Code project" use case.
The MCP tool signature also doesn't accept `library_id`/`library_type` as
arguments — so an agent can't switch libraries dynamically.

**Suggested fix:** add optional `library_id` + `library_type` parameters to
both the MCP tool and `POST /api/zotero/build-kbs/async` request body. Server
falls back to config when omitted.

### H. Zotero KB names use a generic "Library_" prefix — collision risk

Both BioMedOmicsAI's and MetaboLinkAI's Unfiled collections produce
`kb_name: "Library_Unfiled"` (the planner reports `library_name: "Library"`
rather than the actual group name). After running a build, switching
groups and re-running silently merges into / overwrites the prior KB.

**Suggested fix:** in `plan_kbs_from_zotero`, derive the prefix from the
group's actual name via the Zotero `/groups/{id}` endpoint (or from the
`Last-Modified-Version` headers + group meta). Fall back to `Library_`
only for personal libraries.

### O. (deferred) Anthropic-style contextual retrieval

The cheap form of contextual retrieval landed in commit `a2bf676`
(title + section prefix on embeddings). The fuller version uses an
LLM to generate a 50-100 token contextual summary for each chunk
(~30-40% recall lift per Anthropic's benchmarks) but adds ≥ 1 LLM call
per chunk during ingest, which is significant cost for large KBs.
Deferred; recommend gating behind an explicit
`knowledge_base.contextual_retrieval: true` opt-in.

### M. MCP tool gaps surfaced by live audit (9 of 16 tools tested)

Live calls against `localhost:8000/mcp/` via the MCP Python SDK:

- ✅ **`create_knowledge_base`** — works, returns KB metadata.
- ✅ **`add_dois_to_kb`** — works (1 paper / 7 chunks). Same pipeline as
  REST `/api/kb/{name}/dois/async` (but synchronous).
- ⏭️ **`add_papers_to_kb`** — schema accepts `papers: list[dict]`; would
  need full paper-dict shape to round-trip. Untested for that reason.
- ⚠️ **`screen_papers`** — schema declares `candidates: list[str]`, so my
  test using `list[dict]` (with title + abstract) raised
  `string_type` validation errors. Either the docs/description should
  state that `candidates` is plain strings (titles? abstracts?), or
  the signature should accept `{title, abstract}` dicts. As written,
  agents can't infer the right shape without source-reading.
- ⚠️ **`push_to_zotero`** — successfully reaches the Zotero API (cloud or
  local) but returns `{"failed": [{"doi": "...", "reason": "no key
  returned"}]}`. Either the local API doesn't return the key after
  item creation, or the parser path is wrong. Either way the failure
  mode is silent (status 200, no exception).
- ✅ **`ingest_local_documents`** — correctly returns
  `{"error": "local_docs.allowed_roots is empty — server-side path
  ingest is disabled"}` when allowlist is empty. Good guardrail.
- ⚠️ **`build_capsule`** — `kb_name` is a **required** argument despite
  the tool's purpose ("build the capsule for a paper"). The docstring
  should make this explicit so agents don't try it with just
  `paper_id`.
- ✅ **`build_capsules_for_kb`** — works; returns per-paper status. For
  already-built capsules, returns `status: "skipped"` (good — idempotent).
- ❌ **`fetch_paper_resources`** — fails with `paper 'doi:10.1038/...'
  not found in KB 'smoke'` when given the `doi:<doi>` format. The
  REST endpoint and CLI use `doi:` prefix; the MCP tool wants the
  bare paper.id. Inconsistent paper-id format across surfaces.

### L. Zotero ingest cannot reach local-only PDFs — partially mitigated

`zotero_ingest.py:273` calls `download_attachment_bytes(key)` against the
configured `base_url`. With the default (cloud) base_url, attachments that
are NOT uploaded to zotero.org (Linked Files, ZotFile-managed paths, files
parked under a quota cap) are unreachable. The configurable `base_url`
landed in commit `6416fa0` opens the local-API path (`http://localhost:23119/api`)
which serves attachments directly from `~/Zotero/storage`. **However**,
this only helps when the Zotero desktop app's "Local API" is enabled —
Settings → Advanced → "Allow other applications…" — and the user must
have the data locally synced (Zotero's local model). For pure
cloud-only setups this remains a no-op.

### K. Embedding `model_name` shows pipe-separated "primary|fallback"

`FallbackEmbeddingProvider.model_name` returns `"text-embedding-3-small|all-MiniLM-L6-v2"`.
That string is stored on the KB metadata. `check_embedding_compat` then
does string equality across KBs — so multi-KB queries across KBs created
with vs without the fallback wrapper will be rejected as "incompatible".

**Suggested fix:** record only the *primary* model name on the KB, and
log a warning if the fallback ever fires (so the user knows there's
silent embedding-space drift in their data).

## Best-practice notes (good guardrails for next contributors)

1. **`config.yml` is gitignored** (line 105 of `.gitignore`) — safe place
   for `zotero.api_key` etc. **However** the LLM keys come from env vars
   (`ANTHROPIC_API_KEY` etc.). Inconsistent: consider env-var lookup for
   `zotero.api_key` too (e.g. `${ZOTERO_API_KEY}` substitution in the
   config loader).

2. **`.env` is NOT auto-loaded** — no `python-dotenv` call anywhere. The
   `.env` file in the repo is essentially documentation; real keys must
   be in the shell that launches the server (`~/.zshrc`, systemd unit, etc.).
   Either delete `.env` to avoid confusion, or add `load_dotenv()` in
   `cli.py` and document it.

3. **Editable install can drift** — at one point the venv's `.pth` resolved
   `perspicacite` to a worktree under `.claude/worktrees/...`, so edits in
   the main repo had no effect on the running server. After surgery:
   `uv pip install -e . --no-deps` from the repo root to repoint.

4. **Stale `.pyc` caches** can hide source edits — clear `__pycache__`
   when in doubt: `find src -name __pycache__ -type d -exec rm -rf {} +`.

5. **Provider/model in `RAGRequest` has dangerous defaults** (`"deepseek"`).
   Every caller must pass them, or you get the latent bug fixed here.
   Suggest making them `Optional[str] = None` and resolving inside the engine
   from `self.config`.

6. **Two log libraries in flight** — `structlog` and stdlib `logging`. Calling
   one's API on the other's logger raises `TypeError`. Pick one or wrap them.

7. **Anthropic model `claude-3-5-sonnet-20241022` is retired** — `config.example.yml`
   still ships this as default. Update to `claude-sonnet-4-5` (or whichever
   is current at release time) before users hit the same `not_found_error`.

8. **Cheap/dev mode (local everything) is already supported** but not
   documented in one place. To run with **zero API costs**:
   ```yaml
   llm:
     default_provider: "ollama"
     default_model: "llama3.1"           # or mistral, phi3, etc.
     providers:
       ollama:
         base_url: "http://localhost:11434"
         timeout: 120

   knowledge_base:
     embedding_model: "all-MiniLM-L6-v2"   # local sentence-transformers, 384-dim
   ```
   Caveats: (a) Ollama provider has `supports_tools: false`, so agentic
   mode won't work — basic/advanced/contradiction will; (b) existing KBs
   are bound to the embedding model that wrote them — cheap-mode only
   makes sense for fresh KBs. Suggest documenting this in README under
   a "Local-first / cheap mode" subsection.

9. **KB selection per-call via MCP — confirmed working without restart.**
   `generate_report` and `search_knowledge_base` both take `kb_name` as
   an argument. Pattern for "different KB per Claude Code project":
   each project's CLAUDE.md tells Claude to pin a specific `kb_name`
   when calling the shared Perspicacité MCP endpoint. **Note:**
   `build_kbs_from_zotero` does NOT have a `library_id` argument, so
   ingestion is single-library-at-a-time (finding G).
