# E2E real-LLM audit — 2026-05-15

**Scope.** Per the roadmap close-out, run the Perspicacité synthesis
pipeline against two example queries from different domains, on (a)
the direct Anthropic API path and (b) the local `claude` CLI path,
and audit for residual issues.

**Harness.** `tests/audit/run_e2e_audit.py`. Bypasses the disk
cache, feeds 3-paper context per example directly to
`AsyncLLMClient.complete`, captures latency + output + errors to
`tests/audit/results/<provider>-<ts>.json`.

**Examples.**

| ID | Domain | Query |
|---|---|---|
| ex_a | NLP/RAG | "How does retrieval-augmented generation reduce the need for parametric memory in language models, and what are the main architectural variants?" |
| ex_b | Computational biology | "Compare AlphaFold-2, RoseTTAFold, and ESMFold along the axes of speed, accuracy, and reliance on multiple sequence alignments." |

Each example ships 3 paper abstracts from real, citable work.

## Runs executed

| Run | Provider | Model | ex_a | ex_b | Status |
|---|---|---|---|---|---|
| 1 | Anthropic API (direct) | claude-sonnet-4-5 | FAIL (6.4 s) | FAIL (4.5 s) | AuthError × 3 retries each |
| 2 | claude CLI | sonnet | OK (17.6 s, 3351 chars, 11 cites) | OK (16.9 s, 3293 chars, 15 cites) | clean |

Both Claude CLI runs cite all 3 expected DOIs across both examples.

## Findings

### F1. AsyncLLMClient retries on `AuthError` — should fail fast

**Evidence.** The API run (`anthropic_api-1778819931.json`) shows three
sequential `llm_completion_error` log lines per example, separated by
~2 s of backoff. The final error is a typed `AuthError` from
Wave 3.4. Pattern was repeated for both examples — total wasted time
~6 + 4.5 s with ~6 LiteLLM round-trips against an invalid key.

**Impact.** Auth errors are deterministic (the key won't suddenly
become valid). Retrying is pure latency + log noise.

**Fix.** In `AsyncLLMClient.complete`, when the wrapped error is
`AuthError`, skip the backoff loop and raise immediately. Same logic
should apply to a future `ConfigError` class for malformed configs.

```python
except AuthError:
    raise  # don't retry
```

Estimated effort: 5 lines, ~30 min including a unit test.

### F2. agent_cli token usage discarded when no provenance collector

**Evidence.** `_parse_output_with_usage` correctly extracts
`usage.input_tokens` / `usage.output_tokens` (Wave 2.3) and tries to
push them into the provenance collector via
`provenance.context.get_collector()`. If no collector is in the
ContextVar (the audit harness, the CLI `query` command, and probably
any caller not running inside the web app), the data is silently
dropped:

```python
try:
    _c = get_collector()
    if _c is not None:
        _c.add_llm_call(...)
except Exception:
    pass
```

The Claude CLI's own JSON output includes `total_cost_usd`
(observed: $0.030 for a single haiku call); none of this surfaces in
Perspicacité's budget tracker (Wave 2.4) because the path through
`agent_cli.complete` never calls `BudgetTracker.record_cost(...)`.

**Impact.** Wave 2.4 budget caps and per-stage cost accounting are a
no-op for `agent_cli` users. Subscription-mode users have no way to
know they're approaching their share of Claude Code rate limits.

**Fix.** Either:
- (a) Make `AsyncLLMClient.complete` push tokens into
  `BudgetTracker` directly (independent of the provenance collector),
  using an estimated price from `PRICING_TABLE` keyed by
  `(agent_cli, *)`.
- (b) Initialize a default collector when none is present, so the
  data lands somewhere reachable (e.g. an in-memory ring buffer that
  `perspicacite report cost` reads).

Either pulls in ~20–30 lines + a Wave 6.2-style integration test.

### F3. `AuthError` message recommends "wait for quota reset" for invalid keys

**Evidence.** Run 1 surfaced:

> "anthropic: auth failed. Wait for the Anthropic quota reset, or
> route this stage through a fallback provider via
> `llm.providers_per_stage` (DeepSeek / OpenAI / Gemini)."

The error from Anthropic was `invalid x-api-key` — i.e. the key is
malformed/revoked, not over-quota. The message conflates two
different failure modes.

**Impact.** Confusing diagnosis for a first-time user with a typo'd
key. They might wait for a non-existent quota window.

**Fix.** Split `AuthError` recovery hint by sub-pattern:

| Server message | Hint to show |
|---|---|
| `invalid x-api-key` / `unauthorized` | "Your API key is missing or invalid. Set `ANTHROPIC_API_KEY` (or switch to the `claude_code` config preset)." |
| `quota exceeded` / `usage limit` | Keep the current "wait for quota reset" message. |

The regex set lives in `llm/errors.py::_AUTH_PATTERNS`. ~15 lines to
split.

### F4. Claude CLI rich-result fields are lost in agent_cli adapter

**Evidence.** A direct `claude --print --output-format json` call
returns:

```
{"type":"result", "result":"...", "session_id":"...",
 "total_cost_usd": 0.03, "usage": {...}, "modelUsage": {...},
 "duration_ms": 3846, ...}
```

Our `_parse_output_with_usage` keeps only `result`, `usage.input_tokens`,
`usage.output_tokens`. We discard:
- `total_cost_usd` (the CLI's own cost estimate — already net of
  cache / quota).
- `cache_read_input_tokens` / `cache_creation_input_tokens`
  (Anthropic prompt-cache hits vs creates — the right signal for
  whether prompt caching is working).
- `duration_api_ms` vs `duration_ms` (CLI overhead vs actual API
  time).
- `session_id` (useful for cross-call debugging).

**Impact.** We re-estimate cost from `(tokens × PRICING_TABLE)`
when the CLI hands us the exact number for free. Prompt-cache
behavior is opaque — we can't tell whether successive calls are
hitting the cache.

**Fix.** Add a richer return type from `_parse_output_with_usage` —
e.g. `AgentCallResult` dataclass with all six fields — and plumb
through `BudgetTracker.record_cost_direct(cost_usd, cache_hit)`.
~50 lines + a test.

### F5. Output quality differences (sonnet vs haiku) — both acceptable

**Evidence (citation counts, ex_a / ex_b):**

| Run | Char count | Distinct DOIs cited | Total citation hits |
|---|---|---|---|
| sonnet ex_a | 3351 | 3 / 3 | 11 |
| sonnet ex_b | 3293 | 3 / 3 | 15 |
| haiku ex_a | 2656 | 3 / 3 | 6 |
| haiku ex_b | 3402 | 3 / 3 | 10 |

Both models hit all 3 expected DOIs and produce coherent, properly
structured outputs. Sonnet cites more densely (≥2 cites per
paragraph on average); haiku cites once per claim. Both stayed
within the system-prompt instruction to cite by DOI in square
brackets.

Sonnet output read closer to a literature-review tone (paragraphs,
no headers). Haiku output added markdown headers despite no
instruction to do so — minor stylistic divergence, not a bug.

**Conclusion.** No retrieval/synthesis correctness issue surfaced.
Use Sonnet for headline reports, Haiku for fast batched syntheses.

### F6. Latency profile

|---|---|---|---|---|

### F7. Wave 6 perf baseline coverage gap — `synthesis` is mocked

**Evidence.** Wave 6.3's `test_perf_baseline.py` records a
`report_synthesis_seconds` metric, but the metric only times
embedding-cycle overhead. It does not actually run the LLM
synthesizer. So a regression in `complete_with_chain` (e.g.
re-introducing the retry-on-AuthError bug from F1) wouldn't trip
the perf gate.

**Impact.** Wave 6.3's perf gate is blind to the most expensive
stage of the pipeline. The current 17 s sonnet / 7.5 s haiku numbers
are the real "synthesis baseline" the framework should track.

**Fix.** Add a `tests/integration/test_perf_baseline_llm.py`
(separate file because it requires a working LLM provider). Mark it
`live + perf` so CI can opt in via env vars. ~40 lines.

### F8. Embedding fallback is opaque

**Evidence.** Existing KBs in `data/perspicacite.db` have
`embedding_model = "text-embedding-3-small|all-MiniLM-L6-v2"` —
this `|` syntax is the runtime fallback recorded by
`create_embedding_provider`. The audit harness saw it succeed
quietly using `all-MiniLM-L6-v2` because no OPENAI_API_KEY was
present, but a user reading `list-kb` output sees the dual string
with no explanation.

**Impact.** Two failure modes are silently equivalent: "user
intended text-embedding-3-small and we did fall back" vs "user
expected text-embedding-3-small". Re-running a query with a query
embedded under the local model against vectors stored under the API
model would return garbage (dim mismatch or topology mismatch) —
this hasn't been triggered in the audit but is a latent footgun.

**Fix.** Store the *actually-used* model only (single value) and
emit a structured warning log line at the fallback boundary. Capture
the warning in `kb_metadata` row as a separate `embedding_warnings`
JSON field for `list-kb` to surface. ~30 lines.

### F9. Observability dust — LiteLLM "feedback / help" banner pollutes logs

**Evidence.** Every LiteLLM error printed:

```
[1;31mGive Feedback / Get Help: https://github.com/BerriAI/litellm/issues/new[0m
LiteLLM.Info: If you need to debug this error, use `litellm._turn_on_debug()'.
```

This is LiteLLM's stderr noise. Not harmful, but it muddies the audit
artefact and a real operator's terminal.

**Fix.** Silence `litellm` logger at INFO level on import:

```python
import logging
logging.getLogger("LiteLLM").setLevel(logging.ERROR)
```

in `llm/client.py` top-of-module. ~3 lines.

## Recommended action — priority order

1. **F1** (5 min) — stop retrying on AuthError. Pure win.
2. **F2 + F4** combined (~1 h) — make agent_cli budget tracking work
   end-to-end. Unlocks Wave 2.4 for subscription users.
3. **F3** (~20 min) — better diagnostic for missing/invalid API keys.
4. **F9** (~3 min) — silence LiteLLM banner.
5. **F8** (~30 min) — explicit embedding-model recording.
6. **F7** (~1 h) — extend Wave 6.3 perf baseline to cover synthesis.

Everything in this list is mechanical. None touches the Wave 4
scientific features or the Wave 5 MCP surface; this is all
plumbing-layer hardening.

## What worked well

- **Wave 3.4 AuthError surfacing.** The error message did identify
  this as auth (not a generic 5xx), and gave actionable next steps.
  Even though F1+F3 are improvements, the baseline is already
  better than a stack trace.
- **Wave 2.3 token-usage parsing.** The CLI path correctly extracted
  `usage.input_tokens` / `usage.output_tokens` per call — the data
  is there, it just isn't reaching the budget tracker yet.
- **Wave 2.2 embedding cache + 2.1 LLM cache.** Both initialized
  cleanly with bare-config audits, no schema migrations needed.
- **Output citation discipline.** Both Sonnet and Haiku followed
  the system-prompt instruction to cite by DOI in square brackets,
  on the first try, with no prompt iteration. Audit cited every
  source paper in every output.
- **structlog event taxonomy.** `llm_completion_start` /
  `_error` / `agent_cli_call_start` / `_done` made the audit log
  trivially greppable. Future me thanks past me.

## Out of scope

- **Real Anthropic API quality comparison.** Blocked by F1: there
  is no valid `ANTHROPIC_API_KEY` in this environment. Once a
  working key is plugged in, re-run `python tests/audit/run_e2e_audit.py
  --provider api` to capture parity / divergence data.
- **PDF ingest path.** The harness skips the full ingest pipeline
  (DOI → Crossref → PDF → parse → chunk → embed) and feeds
  abstracts directly to the synthesizer. The integration tests
  (Wave 6.1 / 6.2) cover the ingest path with deterministic mocks;
  a real-LLM ingest audit would need 10+ minutes per example for
  PDF downloads and is more useful once F1–F4 land.
- **Multi-KB routing audit.** `auto_route_kbs` is exercised in
  `tests/e2e/test_cross_kb_routing.py` (mocked); a real-LLM audit of
  the BM25-vs-LLM-routing tradeoff is a separate follow-up.
